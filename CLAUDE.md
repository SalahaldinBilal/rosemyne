# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Rosemyne is a ShareX-like screenshot capture/annotate/upload desktop app built with Tauri 2 (Rust backend) and SolidJS + TypeScript (frontend). Windows is the primary target; the Linux capture backend is a stub.

## Commands

- `npm run tauri dev` , run the full app (starts Vite on port 1420 and the Rust backend)
- `npm run dev` , frontend only via Vite (Tauri APIs unavailable)
- `npm run tauri build` , production bundle; `npm run build:debug` , debug bundle
- `cargo check` / `cargo build` inside `src-tauri/` , compile the Rust side alone
- `cargo test` inside `src-tauri/` , run the Rust unit tests

There are no frontend tests or linters configured.

## Architecture

Two Tauri webview windows, both defined at startup:

- **main** (`/` route): screenshot history browser and settings. Created from `tauri.conf.json`; shown/re-created via the tray icon (`focus_or_open_config_window` in `src-tauri/src/lib.rs`).
- **screenshotter** (`/screenshot` route): a borderless transparent window spanning the bounding box of all monitors. Creation/show/hide are platform-specific via the `ScreenshotWindowManager` trait (`src-tauri/src/screenshot_window/`): Windows parks it 32000px off-screen (wraps creation in a per-monitor DPI awareness context, and marks it `WS_EX_TOOLWINDOW` so Chromium/Firefox occlusion trackers keep rendering content behind the live-desktop pick/record overlay instead of pausing and blanking hardware-overlay video), Linux genuinely hides it. If the user destroys it (Alt+F4), `run_callback` in `lib.rs` treats it as a cancelled screenshot and re-creates it.

Screenshot flow: global shortcut or button → `take_screenshot` (`src-tauri/src/screen_manager/commands.rs`) captures the full virtual desktop via GDI (`src-tauri/src/capture/windows.rs`), stores the RGBA image + a WebP preview in `ScreenshotManager`'s in-memory map under a random u16 id, enumerates visible windows (for snap-to-window selection and tagging), then shows the screenshotter window and emits `screenshot://data` to it. The frontend overlay lets the user drag-select a region (or right-click to snap to a window) and add box/text/blur/pixelate overlays , **all overlay rendering is client-side** in `src/helpers/canvasRenderer.ts`: each blur/pixelate canvas composites its region + sampling margin from the capture and the overlays stacked below it (lower effect canvases come from the `effectLayers` registry in `screenshotOverlayState`, invalidated via per-order version counters), so effects stack in z-order with zero IPC during editing. On save, `closeOverlay` hides the window, renders the final image in-page with the same code paths (`renderFinalImage`, on a scene grown by the total effect margin), and sends the raw RGBA through a raw-body invoke (`src/helpers/saveScreenshot.ts`, metadata in the `x-rosemyne-args` header) to `hide_and_save_screenshot`; Rust PNG-encodes, tags visible windows, persists it through the SQLite `HistoryStore`, copies to clipboard, and emits `screenshot://new-saved-image` to the main window. Preview and saved output match by construction.

Overlay attribute defaults (fill/border color, border thickness, text size/font, blur/pixelate intensity) are user-configurable and persisted, under Settings → "Overlay Defaults". `OVERLAY_DEFAULT_ATTRIBUTES` (`src/constants/imageOverlay.ts`) is the built-in baseline; user overrides live in `settings.json` as an `OverlayDefaultOverrides` (Rust: `HashMap<String, HashMap<String, serde_json::Value>>`, values intentionally untyped , Rust doesn't need to understand the attribute schema, only store/return it) via `get_overlay_defaults`/`set_overlay_defaults`. The frontend singleton `src/states/overlayDefaultsState.ts` keeps **one persistent, in-place-mutated store** (`merged`, one entry per overlay type) rather than recomputing/cloning a fresh object on every override change , the Overlay Defaults page binds its controls directly to `merged[type]` slices, and replacing that object wholesale instead of mutating it (e.g. a `createMemo` that re-clones on every change) tears down and rebuilds the DOM under an open color picker, closing it instantly mid-interaction. `defaultAttributesFor(type)` returns an independent snapshot clone for a **newly placed** overlay item (`ImageOverlayContainer.tsx`'s `mouseDownHandler`); placed items are edited afterward via their own per-instance context menu (`ImageOverlayBase.tsx`, `setOverlayItems`), which never touches the shared defaults. Since the screenshotter window is never recreated between captures, `refresh()` re-syncs `merged` from what's actually persisted on every new `screenshot://data` event, so a defaults change made in Settings while the screenshotter was already loaded doesn't leave it stale. The attribute-editing UI itself (`src/components/OverlayAttributeList/`) is shared between the overlay's own right-click menu and the Overlay Defaults page , the color picker's own light/dark text-color heuristic (`@thednp/color-picker`) assumes a translucent color sits over a light checkered backdrop and picks black text whenever alpha ≤ 0.33; Rosemyne renders no such backdrop, so `OverlayAttributeList.module.scss` gives the swatch a flat neutral (`#888`) background instead of the color's own (possibly transparent) one, keeping either text color legible. Box overlays default to a fully transparent fill (`#ff000000`), so a freshly drawn box reads as an outline until the user picks an opaque color.

Video recording flow: the `record` shortcut/button (`record_screen`, or `ShortcutMethod::Record` which toggles stop when a recording is active) shows the same overlay in live record mode (like region-pick: transparent window over the real desktop, no capture) , completing the selection invokes `start_recording`, which hides the overlay, tags visible windows, and starts the platform recorder behind the `ScreenRecorder` trait (`src-tauri/src/recording/`): Windows uses the `windows-capture` crate (Windows.Graphics.Capture frames + hardware Media Foundation H.264/AAC MP4 encoder); Linux is a stub. Region recordings are single-monitor (the monitor under the selection's center, clamped, even-aligned): full-monitor captures go through the zero-copy `send_frame` path, regions through per-frame `buffer_crop` + a row flip (the buffer path wants bottom-up BGRA). System audio is WASAPI loopback via `cpal` (input stream on the default *output* device) feeding `send_audio_buffer`, kept alive through silence by a zero-filled output stream; it degrades to a silent recording if unavailable. Encoding runs to a temp `.recording-*.mp4` in the store base; `stop_recording` finalizes it into history via `HistoryStore::save_recording` (rename + `video` row + first-frame WebP thumbnail + the same derived-tag maintenance as `save_rendered`), then copies the saved file itself to the clipboard per the `copy_to_clipboard_on_capture` setting , via `src-tauri/src/file_clipboard/` (CF_HDROP on Windows, `text/uri-list` through `wl-copy`/`xclip` on Linux), the same module behind the `copy_file_to_clipboard` command used by video/file history cards. While recording, a small always-on-top `recording-hud` window (route `/recording-hud`) shows a timer with stop/cancel, and a click-through `recording-border` ring outlines the region; both are **pre-created hidden at startup** (`create_recording_windows` in `recording/commands.rs`, called from `run_callback`, which also re-creates them if destroyed) so `start_recording` only repositions + shows them , the HUD appears instantly in a "Starting…" state and flips to the timer when the engine is up (the persistent page resets via `recording://overlay-shown`/`-hidden` events, and its status poll blocks on the recording-manager mutex until the in-flight start settles). On Windows both are excluded from capture via `SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)`. Only one recording can be active (`RecordingManagerHandler`).

Data lives in `Documents/Rosemyne/`: `images/`, `thumbnails/` (WebP video thumbnails, named `<fileName>.webp`), `history.db` (SQLite), `settings.json`.

Saved screenshot history is a SQLite store (`src-tauri/src/history_store/`), managed as `Arc<HistoryStore>` (a `Mutex<Connection>`), separate from `ScreenshotManager` (which now only holds in-memory temp captures). The main window loads it **paginated, sorted and filtered server-side** with keyset pagination: `query_history` takes a sort spec (`date`|`name` × `asc`|`desc`) and an optional `{key, id}` cursor where `key` is the sort column's value on the last row (the row count is computed only on the cursor-less first page; name ordering is `COLLATE NOCASE`), and the frontend renders a row-virtualized infinite list (`src/pages/Main/Main.tsx`) that threads `nextCursor` through `loadMore`. The filter tree is **compiled to SQL** (`history_store/compile.rs`): each condition becomes an `IN (SELECT history_id FROM tag_index …)` subquery against the `tag_index` table (one row per scalar tag value, keyed by JSON-stringified path + kind `s`/`n`/`b`), so positive operations (equals, ranges, contains/starts/ends via `instr`/`substr`) never parse tag JSON at query time; conditions on the virtual `$file` paths compile to plain column predicates instead (see Conventions). Only inexact compilations , tag `notEquals` (satisfiable by a missing key) and non-empty `fuzzy` (subsequence match) , append a `filter_match(tags, file_name, file_path, type)` residual: a registered SQLite scalar function (`history_store/filter.rs`) evaluating the `FilterNode` tree per row over the `$file`-augmented tags, a faithful Rust port of `src/states/tagFilterState.ts` (`resolvePath` + `applyOperation`; `fuzzy` is approximated) and the semantic source of truth the compiler's prefilter must stay a superset of. The filter UI's path/type schema is a **cache** in the `meta` (`tag_schema` blob) table read by `get_tag_metadata`; value autocomplete is **queried on demand** via `suggest_tag_values` (case-insensitive substring over `tag_value_counts`, most-common first, LIMIT 50) rather than shipped with the metadata , see Conventions.

Saved capture file names come from the `fileNameTemplate` setting, expanded and sanitized in `history_store/store.rs` (`expand_file_name_template`): `${process}`/`${windowTitle}` (the window covering the largest share of the capture, from the tags), date/time parts, `${width}`/`${height}`, `${random}`/`${random:N}` (N clamped 1–32) and `${guid}`; default `${process}_${random:10}`, `.png` (screenshots) or `.mp4` (recordings) appended, `-1`/`-2`… on collision. Imported files keep their original names , this is a separate setting from where they're placed (below).

Destination **directories** (distinct from file names, above) come from the `uploadPath` ("save path template") setting, expanded by `expand_save_dir` (`history_store/store.rs`, `pub(crate)`): always rooted at `<base>/files/`, with the expanded template as a subpath (`${year}`/`${month}`/`${day}`/`${hour}`/`${minute}`/`${second}`/`${type}`; root/parent path components are stripped so it can't escape `files/`), default `${year}-${month}`. Every history-creating path uses this , fresh captures/imports (`import_file`) and ShareX migration (`sharex_migration/migration.rs`) alike , so imported files land organized the same way a fresh capture would, instead of one flat dump folder. ShareX migration computes the directory **per row from that row's own original ShareX date**, not the migration's run time, so old screenshots stay organized under their original period instead of all bucketing into the current month.

Images are served to the frontend through the custom `rosemyne-photo` URI scheme registered in `lib.rs`: `http://rosemyne-photo.localhost/saved/<fileName>` (from disk; honors `Range` requests with 206 chunks so `<video>` streams and seeks without loading whole files), `/thumb/<fileName>` (WebP video thumbnail from `thumbnails/`, 404 when absent) and `/preview/<id>` (in-memory WebP preview , also the bitmap source for all canvas rendering). `/saved` and `/preview` send `Access-Control-Allow-Origin`, so canvases stay untainted (`getImageData`, and the video-thumbnail generation in `src/helpers/videoThumbnail.ts`).

Video thumbnails are always generated **client-side**, never by Rust: `generateVideoThumbnail`/`captureVideoFrame` (`src/helpers/videoThumbnail.ts`) load the video into a hidden `crossorigin` `<video>` element, seek to an early frame, draw it to a canvas, and `canvas.toBlob(..., "image/webp")` , the browser's own WebP encoder, no server-side encoding involved , then ship the bytes via the raw-body `save_video_thumbnail` command (`HistoryStore::save_thumbnail_bytes` just writes them to `thumbnails/` as-is). Two call sites: `Main.tsx`'s video cards generate lazily when their thumbnail `<img>` 404s (i.e. only for videos the user actually scrolls past), and `ShareXImport.tsx` also generates proactively right after a real import for every video at least `LARGE_VIDEO_BYTES` large still missing one (`list_videos_missing_thumbnail`/`HistoryStore::videos_missing_thumbnail`), so a big migration doesn't leave large videos stuck on a placeholder indefinitely.

Screenshot/video cards (`Main.tsx`) support dragging the card **out to another app** as a real OS file drag via `tauri-plugin-drag`'s `startDrag` (not HTML5 drag-and-drop): the dragged item is the card's actual saved file, with a small (≤128px) preview icon rendered server-side by `HistoryStore::drag_icon_path`/`get_drag_icon` (from the image itself, or the video's thumbnail if it has one; falls back to no custom icon , just the OS default cursor , for videos without a thumbnail yet or plain files, since Windows' image decoder can't preview those anyway). Dropping a card back onto the app's own window is a no-op, not a duplicate: `HistoryStore::import_file` checks whether the source path already lives under the store's own base directory and returns `None` without copying/recording if so.

Frontend↔backend calls go through `safeInvoke` (`src/helpers/safeInvoke.ts`), which types every Tauri command name, its args, and return type in the `Commands` map. When adding or changing a `#[tauri::command]`, register it in the `invoke_handler` list in `lib.rs` **and** update the `Commands` type. Rust structs are serialized `camelCase` (serde rename) to match the TS types in `src/types/`. One exception: `hide_and_save_screenshot` takes a raw binary body (`tauri::ipc::Request`) and is invoked via `src/helpers/saveScreenshot.ts` instead of `safeInvoke`.

Global state on the frontend uses the `createRoot(useXStateInner)` singleton pattern (`src/states/`). The `@core` Vite alias maps to `src/`.

## Conventions

- Emitting events from Rust to the frontend must go through the `emit_on_main_thread!` macro (defined in `lib.rs`) , emitting cross-thread can deadlock with WebView2 on Windows (tauri-apps/tauri#9453, #11787). Never call `.emit()` directly.
- Tagged Rust enums exposed to the frontend use `#[serde(tag = "type", content = "data", rename_all = "camelCase")]`; mirror them in TS with the `BaseUploaderType`-style discriminated unions in `src/types/request.ts`. **The enum-level `rename_all` only cases the variant tag, not fields inside a struct variant** , a multi-word field name on a struct variant (e.g. `response_headers`) is serialized as-is unless that variant also gets its own `#[serde(rename_all = "camelCase")]` (see `UploaderBodyHandler::FormData`, `OverlayInstruction::Box`). Forgetting this silently desyncs the JSON from the TS type instead of failing to compile , always add the per-variant attribute on any struct variant with a multi-word field.
- **Every mutation of the `history` table must keep the derived tag tables in sync.** There are two: the tag-metadata cache (`tag_schema` blob in `meta` + `tag_value_counts`, read by `get_tag_metadata` instead of scanning `history`) and the `tag_index` query-acceleration table (read by the compiled filter in `compile.rs`). Any insert/delete that touches a row's `tags` must, in the same operation, merge the schema, increment/decrement the value counts (prune rows hitting 0), and insert/delete the row's `tag_index` entries. Route all history writes through `HistoryStore` methods (`save_rendered`, `insert_batch`, `delete_with_file`) , they already do this; don't write the `history` table directly. Both are derived indexes: they are fully rebuilt by a one-time scan on open whenever `meta.metadata_status` isn't the current `METADATA_STATUS_BUILT` value (the status is set to `building` before the scan so an interrupted rebuild is detected and restarted). If you add a new history-mutating path, change the tag shape, or change what the derived tables store, update the maintenance alongside it and bump `METADATA_STATUS_BUILT` to force a rebuild.
- `tag_index` and `filter_match` (`filter.rs`) implement the same filter semantics twice: the compiled SQL must match a **superset** of what `filter_match` matches (exactly equal when `CompiledFilter::exact`). `collect_index_entries` in `compile.rs` must flatten tags exactly like `resolve_path` in `filter.rs` resolves them (arrays transparent, objects extend the path, nulls dropped, booleans kept). When touching filter semantics, update both and extend the `compiled_queries_match_reference_eval` parity test in `store.rs`.
- **Every derived reactive value in a SolidJS component must be wrapped in `createMemo`**, never a plain `const x = () => …` arrow function , this holds regardless of how cheap the computation is, not just for expensive/multi-read cases. This applies whether the reactive dependency is closed over (a signal/prop/store field read directly in the body) or passed in as an argument that is itself reactive , a function called with a reactive value (e.g. `screenshot.fileName` from a store row) needs its call site memoized just as much as one with no arguments at all. The only real exemptions: plain functions with no reactive reads inside them at all (pure, generic, stateless), and imperative helpers only ever invoked from event handlers/async flows and never read reactively in the render tree (e.g. `currentFilter()` in `Main.tsx`).
- **Top-level tag keys starting with `$` are a reserved system namespace.** The only current member is the virtual `$file` tag (`["$file","Name"|"Path"|"Type"|"DateTime"|"Size"]`), which exposes the `file_name`/`file_path`/`type`/`date_time_ms`/`file_size` columns through the normal filter machinery: `get_tag_metadata` injects its schema at read time (`Size`'s schema `type` is `"byteSize"`, a frontend-only kind rendering `FileSizeField`, a unit-picker like `DurationField`'s), `compile.rs` (`file_column`) turns its conditions into column predicates, `filter.rs` (`augment_tags`) inserts a `$file` object into each row's parsed tags before residual evaluation, and `suggest_tag_values` special-cases its autocomplete (numeric fields , `DateTime`/`Size` , return no suggestions). Every operation except `fuzzy` compiles exactly for the always-present text columns and `DateTime`; `Size` (backed by nullable `file_size`) is treated the same as always-present for simplicity, so `notEquals` on a row with no recorded size (a rare failed-`stat` case in `save_recording`) is a known, accepted narrow divergence from the reference evaluator rather than specially handled. `filter.rs` (`augment_tags`) omits `Size` entirely (not `null`) when `file_size` is absent, so it resolves as a missing key like any other optional tag. The filter UI renders `$`-keys prettified (`$file` → "File", `keyLabel` in `FilterConditionView`). `$file` entries are never written to `history.tags` or `tag_index` , they're synthesized at query time. If user-defined custom tags are ever added, reject keys starting with `$` at that boundary so collisions stay impossible.
